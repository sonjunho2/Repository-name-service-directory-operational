# 회원관리 닉네임 저장 오류 정밀 수정 보고서

## 확인된 원인

1. 관리자 회원관리 화면에 기존 일반 form 방식(`/admin/user`)과 AJAX 방식(`/admin/users/:id/update`)이 혼재할 수 있는 구조가 남아 있었다.
2. AJAX 저장 라우트에서 `nickname` 값이 비어 있거나 누락되면 서버가 기본값 `회원`으로 저장하도록 되어 있었다.
3. 이 때문에 프론트에서 값 전달이 한 번이라도 누락되면 운영자가 입력한 닉네임 대신 `회원`으로 덮어쓰이는 위험이 있었다.
4. 회원 행 렌더링에서 업체회원의 필터용 `data-role`이 실제 표시값과 다르게 처리될 수 있어 권한/계정구분 표시가 혼동될 수 있었다.

## 수정한 내용

### views/admin.ejs

- 회원관리 테이블의 기존 form 기반 저장 구조를 제거하고, 단일 AJAX 저장 버튼(`.user-save-btn`) 구조로 통일했다.
- 닉네임/권한/상태/비밀번호 입력값은 같은 행의 입력 필드에서만 수집하도록 유지했다.
- 저장 전 닉네임이 비어 있으면 서버 요청을 보내지 않고 즉시 오류 알림을 표시하도록 보강했다.
- 계정구분 표시와 필터 기준을 `admin / vendor / user`로 일치시켰다.
- 회원관리 저장 함수는 1개(`saveUser`)만 사용한다.
- 기존 `userForm*` / `form="userForm*"` 구조는 제거했다.

### server.js

- `/admin/users/:id/update` 라우트에서 닉네임 누락 시 `회원`으로 덮어쓰지 않도록 수정했다.
- 닉네임이 빈 값이면 JSON 요청에는 명확한 오류 메시지를 반환한다.
- 기존 닉네임을 불필요하게 초기화하지 않도록 현재 사용자 정보를 먼저 조회한다.
- 관리자 계정의 관리자 권한 해제는 계속 차단한다.
- `/admin/user` 레거시 POST 라우트도 동일하게 닉네임 빈 값 저장을 막도록 보강했다.

## 검증 결과

- `node --check server.js` 통과
- Express 라우트 중복 0건 확인
- `views/admin.ejs` 내 회원 저장 함수 `saveUser` 1개 확인
- `views/admin.ejs` 내 기존 `userForm` 참조 제거 확인
- 회원 저장 fetch URL `/admin/users/:id/update`와 Express POST route 일치 확인

## 배포 후 테스트 순서

1. Render 재배포 후 `/healthz` 접속 확인
2. 관리자 로그인 후 `/admin#users` 이동
3. `qqq111` 회원 닉네임을 `qqq111` 또는 다른 값으로 변경 후 저장
4. 새로고침 후 닉네임이 입력값 그대로 유지되는지 확인
5. 권한을 일반회원/관리자로 변경 저장 테스트
6. 상태를 정상/차단으로 변경 저장 테스트
7. 새 비밀번호 6자 미만 입력 시 오류 처리 확인
8. 업체회원 연결 후 계정구분이 업체회원으로 표시되는지 확인

## 남은 주의사항

- 권한 select는 로그인 권한(`user/admin`)만 변경한다. 업체회원 여부는 별도 `회원 → 업체회원 연결` 기능에서 `is_vendor/vendor_id`로 처리한다.
- 운영 DB에서 이미 `회원`으로 덮어써진 닉네임은 원래 값을 서버가 알 수 없으므로 관리자 화면에서 다시 입력해 저장해야 한다.
